# 文档治理规则

## 1. 基本哲学

### 文档是变更协议，不是代码副本

文档说明意图、边界、决策和验收，不逐行复述实现。能够从代码直接读出的细节不重复维护。

### 先写最小记录，再动实现

变更记录不是长篇设计书。开始工作前只需要把目标、边界、风险和验收写清楚；实施后再补实际结果。

### 小文件优于大百科

每个文件只回答一个稳定问题。不同生命周期的信息不得混放：领域现状、单次变更、长期决策分别管理。

### 索引只导航，不承载正文

索引文件只放 ID、标题、状态和链接。任何解释性正文放在对应记录中。

### 历史不可伪装成现状

完成的 Change Record 是历史证据；领域文档描述当前事实；ADR 描述仍然有效的长期决策。

## 2. 文档类型

| 类型 | 回答的问题 | 生命周期 |
|---|---|---|
| Domain | 这个领域现在是什么样 | 持续更新 |
| Roadmap | 这个领域接下来按什么里程碑推进 | 只保留活跃和近期目标 |
| Epic | 一个复杂目标如何拆成可独立验收的变更 | 完成后冻结主体 |
| Change | 为什么要进行这一次改动 | 每次变更一份，完成后冻结主体 |
| ADR | 为什么长期采用这个方案 | 接受后稳定，废弃时新建替代 ADR |
| Registry | 当前需要读哪些文档 | 持续维护且保持短小 |

## 3. 标准变更流程

1. 从模板创建 `changes/YYYY/MM/<change-id>.md`。
2. 在 `registry/ACTIVE.md` 增加一行。
3. 填写目标、非目标、影响、边界和验收，状态为 `proposed`。
4. 需要共同评审的 Contract 或架构改动先完成评审，再改为 `approved`。
5. 开始代码修改时改为 `in_progress`。
6. 验证后填写结果、偏差和后续项，改为 `completed`。
7. 从 `ACTIVE.md` 删除该行；历史通过年月目录定位，不汇总到无限增长的总表。
8. 若改变稳定现状，更新对应 Domain；若形成长期选择，新建 ADR。

## 4. 复杂任务模型

复杂目标使用一份 Epic 管总目标，多份 Child Change 管独立实施：

```text
Roadmap Milestone
└── Epic
    ├── Child Change A
    ├── Child Change B
    └── Child Change C
```

- Epic 只记录总体目标、边界、子任务关系和总体验收，不承载实现细节。
- Child Change 必须可独立验收，使用 `parent` 指向 Epic。
- `depends_on` 表示开始前必须完成的任务；`blocks` 仅作反向导航，不能代替 `depends_on`。
- 一个 Change 只能有一个 `parent` 和一个主领域，可以关联多个 ADR。
- Roadmap 管理里程碑，不列出全部历史 Change；完成里程碑应归档或压缩为结果链接。

满足以下任一重要条件时，应拆成 Child Change 或 ADR：

- 涉及两个以上可独立演进的 package；
- 包含两个以上可独立验收或回滚的结果；
- 存在不同负责人、依赖或审批边界；
- 同时包含长期技术决策与具体实现；
- 单文件接近 250 行或 20 KiB；
- 一个子部分阻塞时，其他部分仍可继续。

禁止通过 `part-1`、`part-2` 机械切割；拆分后的文件必须具有独立语义。

## 5. 状态定义

```text
proposed → approved → in_progress → completed
    └──────────────→ rejected
approved/in_progress → blocked
blocked → in_progress/rejected
```

- `proposed`：只提出，不能开始修改代码。
- `approved`：边界已确认，可以实施。
- `in_progress`：正在修改。
- `blocked`：存在明确阻塞条件。
- `completed`：实现和验证完成。
- `rejected`：不再实施，保留原因。

纯文档勘误可由作者自批准；Contract、公共 API、跨 Feature 或架构边界变更必须按仓库现有评审规则审批。

## 6. 文件大小控制

- 建议单文件不超过 300 行或 20 KiB。
- 达到 250 行时必须评估拆分。
- 超过上限时按子领域或独立决策拆分，不能仅建立 `part-1`、`part-2`。
- Change Record 不追加无界日志；最多保留关键实施结果和链接。
- 领域文档的历史内容迁移到 Change/ADR，Domain 只保留当前事实。

## 7. 所有权与冲突

- 一个变更记录只能有一个主负责人。
- 并行任务使用不同 Change ID，不共同追加同一文件。
- 跨领域变更指定一个主领域，其他领域只记录链接，避免复制正文。
- 文件创建后不因状态变化移动；状态只更新 Front Matter 和 Active 索引。
- 契约冲突以 `contracts/` 为最终真相源；文档不得覆盖正式 Contract。

## 8. 禁止事项

- 用一个 `CHANGELOG.md` 承载所有实施细节；
- 每次任务递归读取全部历史；
- 在多个领域文件复制同一设计；
- 把计划、会议纪要、当前架构和调试日志混在一个文件；
- 修改历史 Change Record 来掩盖实施偏差；
- 只创建记录但不填写验收结果。
- 使用 `pending/in-progress/completed` 目录移动记录；
- 建立包含所有历史任务的无限增长总执行队列。
